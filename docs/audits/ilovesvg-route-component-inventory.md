## Method and evidence rules

Verified facts come from app/routes.ts, app/data/routeManifest.ts, app/data/routeManifest.types.ts, public/sitemap.xml, public/robots.txt, app/client/components/navigation/toolNavSections.ts, route source files, wrapper imports, shared converter components, shared settings/preset modules, and existing read-only audit scripts. “Inference” and “runtime verification required” are used where a rendered state, conditional control, or behavioral difference cannot be proved from static source.

The route manifest contains no h1 values. H1 values below were taken from route JSX, CurrentRouteTitle inheritance, or the shared documentation hero content. Wrapper rows use the route-aware CurrentRouteTitle label supplied by app/client/components/navigation/OtherToolsLinks.tsx. Metadata comes from the route-local meta function or the manifest-backed route-meta factory. “Schema none found” means no JSON-LD or schema.org markup was found in the effective component source; it does not mean the application shell cannot add unrelated site-wide schema.

Internal-link evidence uses the manifest nav/related policy plus app/client/components/navigation/toolNavSections.ts. The navigation audit verified 123 section hrefs, five primary hrefs, zero missing targets, zero redirect targets, and zero duplicate section hrefs. The homepage and sitemap account for the two indexable routes outside those 123 section hrefs. The three legal routes are non-indexable and remain linked through site/footer or HTML-sitemap paths. Redirect aliases are intentionally absent from navigation.

## 1. Executive summary

The repository registers 157 routes: 156 public routes and one non-public API/action route. Of the public routes, 125 are indexable and present in the XML sitemap, 28 are non-indexable redirect aliases, and three legal routes are HTML-sitemap-only. The XML sitemap and the manifest’s XML policies match exactly: 125 unique URLs, no missing manifest URL, and no extra XML URL.

The largest consolidation risk is not converter logic but ownership of route intent. Fifty-plus small wrapper files inherit a complete converter component and often the same accepted inputs, outputs, presets, settings, preview, and export workflow. Their differentiation is primarily metadata, CurrentRouteTitle, CurrentRouteGuide, and selected pathname-specific content. That makes them plausible future merge candidates, but not redirect-safe now: unique platform/format guidance, internal links, schema identity, and real browser behavior must be preserved first.

Confirmed source defects or risks found in this audit:

- The favicon component’s visible breadcrumb and BreadcrumbList schema point to /svg-favicon-generator, which is not a registered route; the registered canonical is /svg-to-favicon-generator. All favicon/ICO wrappers inherit this component.
- SVG Stroke Width Editor links and schemas to /svg-stroke-width-adjust, and SVG Flip & Rotate links and schemas to /svg-flip-rotate-editor; neither path is registered.
- Several wrapper families inherit base-component schema URLs rather than their self-canonical route: SVG-to-PNG, SVG-to-JPG for Etsy, SVG resizer, JPG platform wrappers, PNG platform wrappers, transparent-PNG, and favicon/ICO wrappers. This is a schema identity mismatch, not a canonical-tag mismatch.
- /image-to-svg-converter performs a verified 301 to /, but its manifest row omits redirectTo. The actual loader and canonical are correct; the manifest inventory is incomplete.
- app/data/routeManifest.ts omits title and description fields for 11 indexable routes even though each route-local meta function supplies both. This is a source-of-truth drift risk, not a live metadata absence.
- app/routes/sketch-to-svg-converter.tsx contains a clearly unrelated public rental/budgeting disclaimer.
- Public conversion state can expose VTracer, Potrace, “backend processing,” “pipeline,” and “parity” implementation language.
- Route-guide copy includes public editorial phrases such as “PNG to SVG keyword cluster” and “search intent.”
- Local preset IDs collide with shared addition IDs in six route sources. PresetSelector deduplicates by id + label + effective settings, not by id alone; both entries can therefore survive while ID-based active/pin/label lookup remains ambiguous. Runtime verification is required before calling the user behavior broken.

Normal and expected structure:

- Direct raster/Cricut/layered converters are mixed client/server workflows: route action validation and Potrace/Sharp helpers coexist with optional browser VTracer, local SVG editing, output history, copy, and download.
- SVG raster/PDF/favicon and most SVG utilities are client-only and intentionally use different settings and output models.
- Dimensions Inspector and File Size Inspector are functionally distinct.
- Cricut SVG Converter and Code to SVG for Cricut accept different source representations and are not duplicates.
- No duplicate indexable manifest title or meta description was found. No XML sitemap/canonical-policy disagreement was found.
- No redirect is newly recommended as safe in this pass.

## 4. Route-family analysis

### Favicon family

The strongest functional destination is /svg-to-favicon-generator because its single client component accepts SVG, PNG, JPEG, and WebP and emits multi-size ICO, PNG icon assets, HTML snippets, optional manifest/browserconfig content, individual downloads, and ZIP. /image-to-favicon-generator, /png-to-favicon-generator, /jpg-to-favicon-generator, /logo-to-favicon-generator, /svg-to-ico-converter, /png-to-ico-converter, /svg-to-favicon-for-shopify, and /logo-to-favicon-for-shopify inherit that component and settings.

Actual difference: route metadata, CurrentRouteTitle, route guide/platform intent, and some wrapper-level SEO content. Confirmed shared defect: the inherited visible breadcrumb and BreadcrumbList schema use /svg-favicon-generator, an unregistered path. Redirect currently safe: no. Browser verification must cover each named-input wrapper accepting the inherited broader formats, generated decoded previews, ICO contents, ZIP, snippets, clear, second input, and unmount/download initiation.

### PNG/platform family

/png-to-svg-converter is the strongest general PNG destination. /png-to-svg-for-canva and /png-to-svg-for-figma inherit it directly. /png-to-svg-for-shopify inherits /png-to-svg-for-etsy; Shopify-specific pathname content exists in that template. The platform routes inherit real converter behavior, not a reduced form: PNG/JPEG/WebP-family acceptance, mixed trace processing, presets, advanced settings, bounded output history, layer editing, copy, download, reset, and second-upload behavior.

Manifest family says cricut-craft for these platform routes while getRouteCapabilities returns raster-to-svg. That does not prove a runtime bug, but it is a taxonomy/source-of-truth mismatch. Redirect currently safe: no.

Evidence: app/routes/png-to-svg-converter.tsx, app/routes/png-to-svg-for-etsy.tsx, wrapper files, app/client/lib/converter/routeCapabilities.ts.

Confirmed risk: JsonLdBreadcrumbs always identifies /svg-to-png-converter, so wrapper schema disagrees with each wrapper’s self-canonical. The converter changes raster background, not arbitrary SVG fill/stroke colors; static source did not prove a misleading recolor control, but platform wording and live controls require browser verification. /svg-resize-and-scale-editor is the strongest functional destination. Their inputs overlap, but outputs, settings, reports, and user tasks do not. Strong canonical destination: each self. Redirect currently safe: no.

### Cricut/code family

/cricut-svg-converter is a raster/upload-oriented mixed trace converter with Cricut presets and history. /code-to-svg-for-cricut accepts SVG code, data URIs, and embedded supported image data, then sanitizes or traces depending on source. /base64-to-svg-for-cricut accepts encoded SVG or raster data with decode/cleanup controls. These routes share tracing helpers but have unique inputs, parsing, warnings, and content. Redirect currently safe: no.

### Sketch/drawing family

/sketch-to-svg-converter and /drawing-to-svg-converter both use mixed raster tracing and the shared advanced/output architecture, but their local presets, defaults, accepted-input copy, source guidance, and route content differ. /sketch-to-svg-for-cricut is layered/cut-oriented and uses LAYERED additions; /drawing-to-svg-for-cricut is a separate cut route with different presets. A confirmed unrelated rental/budget disclaimer appears only on the general sketch route. Redirect currently safe: no.

### JPG/JPEG family

/jpg-to-svg-converter and /jpeg-to-svg-converter are the closest pair among direct, non-wrapper converters. Both support mixed tracing, history, editing, and broad presets, but their accepted input sets and local preset inventories are not identical: JPG accepts JPG/JPEG/PNG/WebP/SVG; JPEG’s source advertises the broader PNG/JPG/WebP/GIF/AVIF/BMP/TIFF/SVG family and includes an additional ink preset. /jpg-to-svg-converter is prominent in primary navigation, but repository evidence alone cannot establish traffic or a safe canonical winner.
